      What shipped (diverged from the original plan - see below):
      SQLite (DATA_DIR/habits.db) with a wide `daily_entries` table -

      simplicity at this scale. Fields (config-driven via HABIT_FIELDS
      in main.py, no longer TBD): sleep_hours (number), notes (optional
      text) - type-based validation generalized in commit 50f2e75.

      API shipped: GET /api/habits/fields, GET /api/habits/today,
      POST /api/habits (upsert), DELETE /api/habits/today (edit flow,
      commit 95eb6c7). The originally-planned GET /api/habits?start=&end=
      range-query/stats endpoint was dropped - 2026-07-22 decision to
      explore the data directly in Jupyter instead of building a stats
      API/view.

      Reminder tie-in: /api/jobs/daily-check skips the push if today's
      already logged. DONE 2026-07-19. Had a real bug where the "today"
      check used server UTC instead of US/Eastern, so logging late
      evening could pre-emptively mark the next day as done - fixed
      2026-07-22 (commit b3106ec) via zoneinfo(America/New_York).

      In-app prompt: DONE. index.html shows a modal built from
      /api/habits/fields when /api/habits/today reports not logged, with

      logged.

      Auth question resolved 2026-07-23 - see "Auth" under UI Design
      Decisions below. /api/habits/* now requires a logged-in session,
      same site-wide gate as everything else.

UI Design Decisions TODO
==========================
Raised 2026-07-21, ahead of building out multiple screens (settings,
stats, etc.) beyond the current single index.html page. Decided:
multi-screen navigation will use separate Flask routes/templates (real
URLs, back button, bookmarkable) rather than show/hide divs in one page.

- [ ] Shared layout. Pull common chrome (styles, nav, SW registration
      script) into a Jinja base template (`{% extends %}`) instead of
      copy-pasting the <style> block and boilerplate JS into every new
      page.

- [ ] Service worker shell list. static/sw.js's SHELL array and
      cache-first strategy only cover '/' right now. New routes like
      /settings need to be added to SHELL (for offline access) or
      deliberately left network-only. Remember: any template change
      still means bumping CACHE, same as every habit-modal change did
      this session.

- [x] Auth. DONE 2026-07-23. Single shared password (SITE_PASSWORD env
      var), checked via a login form that sets a signed Flask session
      cookie (SECRET_KEY env var, 30-day lifetime). GET /login and
      POST /login (login.html), POST /logout. A before_request hook in
      main.py gates everything except /login, /healthz, /sw.js,
      /static/*, and the two X-Cron-Secret-protected job endpoints
      (/api/notify, /api/jobs/daily-check - those stay header-authed
      since they're hit by cron, not a browser session). This also
      resolved the /api/habits/* auth question above.

      Known gap, not fixed: static/sw.js caches '/' cache-first. After
      a logout on a device that previously cached '/' while logged in,
      that cached copy can still be served offline instead of hitting
      the network and redirecting to /login. Not addressed - fine for
      a single-user hobby app, but worth knowing if this ever needs to
      hold more sensitive data.

- [ ] Where settings live. Server-side in SQLite (like habits.db -
      follows you across devices, needs a table/endpoints) vs.
      client-side in localStorage (zero backend, but per-device/
      per-browser only, same limitation as the current push
      subscription).

- [ ] Mobile-first chrome. Pick one navigation pattern (bottom nav for
      thumb reach vs. top nav) and account for safe-area insets
      (notches) and touch target sizing once, rather than per-screen.
